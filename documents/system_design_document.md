# Real-Time Order Lifecycle Data Platform

## 1. Overview
The **Order Lifecycle Data Platform** is an event-driven data system designed to capture, process, and serve real-time order-related events generated by multiple backend services in an e-commerce environment.

---

## 2. Objectives
### Primary Goals
* **Capture events in real-time:** Real-time order tracking.
* **Reliability:** No data loss or duplication.
* **Dual Storage:** Maintain raw and processed data stores.
* **API Delivery:** Operational and analytics APIs.

### Secondary Goals
* Microservices simulation.
* Idempotency and retries.
* Interview-grade system visibility.

---

## 3. System Scope
* **Included:** Event generation, Kafka bus, Stream consumer, Raw/DB storage, FastAPI.
* **Excluded:** Spark, Airflow, Cloud-specific deployment, Multi-node Kafka.

---

## 4. High-Level Architecture
1. **Order/Payment/Delivery Services** → Event Producers.
2. **Kafka** → Event Bus.
3. **Python Consumer** → Stream Processing.
4. **PostgreSQL + JSON Files** → Storage Layer.
5. **FastAPI** → API Serving Layer.

---

## 5. 📡 Data Sources
* **Order Service:** Creation and cancellation.
* **Payment Service:** Success and failure lifecycle.
* **Delivery Service:** Fulfillment and tracking.

---

## 6. 📦 Event Model
```json
{
  "event_id": "uuid",
  "order_id": "string",
  "user_id": "string",
  "event_type": "order_created | payment_success | order_delivered",
  "timestamp": "ISO-8601",
  "service": "order | payment | delivery",
  "status": "success | failed",
  "metadata": {}
}
```

---

## 7. ⚙️ Kafka Design
* **Topic:** `order-events`
* **Partition Strategy:** Keyed by `order_id` (ensures ordering per order).
* **Consumer Model:** Single group, at-least-once processing.

---

## 8. 🧪 Stream Processing Layer
* **Validation:** Schema and type checking.
* **Deduplication:** Idempotency via `event_id`.
* **Transformation:** Structured records for SQL.
* **Routing:** Dual-write to audit logs and PostgreSQL.

---

## 9. 💾 Storage Design
### 9.1 Raw Data Layer (Audit)
* JSON file-based storage for replay/debugging.

### 9.2 Structured Storage (PostgreSQL)
**Table: `orders`**
* `order_id` (PK), `user_id`, `current_status`, `updated_at`.

**Table: `order_events`**
* `event_id` (PK), `order_id` (FK), `event_type`, `timestamp`, `service`.

---

## 10. 🌐 API Layer (FastAPI)
### Operational APIs
* `GET /order/{id}`: Current status.
* `GET /order/{id}/history`: Timeline of events.

### Analytics APIs
* `GET /metrics/orders/count`: Total orders.
* `GET /metrics/orders/status-breakdown`: Aggregate stats.

---

## 11. 🔐 Reliability Design
* **Idempotency:** Prevent duplicates using unique event IDs.
* **Retry:** Transient failure handling in consumer.
* **Ordering:** Kafka partitioning per order.
* **Failure:** Dead Letter Queue (DLQ) concept.

---

## 12. 📊 Observability
